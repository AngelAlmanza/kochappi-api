# ADR-001: Use Integer IDs Instead of UUIDs for Entity Primary Keys

**Date:** 2026-03-18
**Status:** Accepted

---

## Context

The initial implementation used UUID (v4) strings as primary keys for all entities, including the `users` table. This is a common pattern for distributed systems or applications with high user counts, where sequential IDs could expose business metrics or allow enumeration attacks.

However, Kochappi is a personal training management tool designed for a single coach and a small, closed group of customers. The system does not expect more than ~50 users at any point.

## Decision

**All entity primary keys now use auto-incrementing integers (`SERIAL` / `INTEGER`), except `refresh_tokens.id` which remains a UUID.**

### Why integers for entity IDs

- **Simplicity:** Integer keys are simpler to work with in queries, logs, and debugging.
- **Performance:** Integer comparisons and joins are faster than UUID string comparisons.
- **Predictability at this scale:** With fewer than 50 users and no public registration, enumeration attacks are not a practical threat. Access is controlled by role-based authentication.
- **No distribution concerns:** The system runs on a single PostgreSQL instance with no need for globally unique IDs across nodes.

### Why `refresh_tokens.id` stays as UUID

Refresh token IDs are security-sensitive and must be **unpredictable**. An attacker who can guess or enumerate token IDs could replay a session without knowing the user's credentials. Sequential integers would make this trivial.

UUID v4 (randomly generated) provides ~122 bits of entropy, making brute-force or enumeration attacks computationally infeasible. This is a well-established security practice for session tokens, CSRF tokens, and similar credentials.

The `refresh_tokens.id` is never exposed as a URL parameter or in business logic — it exists solely as an opaque, unguessable identifier stored in a JWT claim and verified against the database.

## Consequences

- The `users.id` field is now `int` (auto-incremented by PostgreSQL).
- The `refresh_tokens.user_id` foreign key is now `int` (matching `users.id`).
- `refresh_tokens.id` remains `varchar(36)` / UUID, generated by the application using `uuid.New()`.
- JWT claims (`user_id`) now carry integer values instead of UUID strings.
- The `UserResponse.id` field in API responses is now an integer.
- The `GET /auth/me` response `user_id` field is now an integer.
